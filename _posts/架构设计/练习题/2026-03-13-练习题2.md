# 问题8  
## Redis扣减库存成功，但MQ发送失败会不会库存丢失？

### 我的回答
1. Redis扣减库存成功后，DB未扣减库存，实际库存未减少，但缓存库存减少  
2. 原因：MQ消息发送失败，订单服务未消费消息，DB未扣库存  
3. 解决：发送MQ时同时保存DB可靠消息，通过定时任务扫描重试发送  

缺点：依赖可靠消息定时任务扫描

### 更优回答
会出现库存丢失问题：

Redis扣库存成功 → MQ发送失败 → 订单服务未消费 → 未创建订单  

解决方案：**本地消息表（可靠消息）**
流程：
1. 发送MQ前写入消息表
2. 发送MQ
3. 若发送失败，通过定时任务重试
4. 保证MQ消息最终投递成功

### Redis与DB一致性
无法保证强一致，只能保证：
**最终一致性**

# 问题9  
## MQ消费成功，但DB写订单失败怎么办？

### 我的回答
1. 消费端支持重试：DB失败回滚事务，返回失败让MQ重新投递  
2. 幂等处理：秒杀活动ID + 用户ID判断是否已创建订单  

### 更优回答
1. **MQ重试机制**  
   - DB失败 → 回滚事务 → 返回失败 → MQ重新投递
2. **幂等处理**  
   - activityId + userId 查询是否已下单
3. **数据库唯一键兜底**  
   - activityId + userId 建唯一索引  
   - insert重复时捕获异常
4. **补偿处理**  
   - 超过最大重试次数 → 死信队列  
   - 告警 + 人工处理

# 问题10  
## Redis扣库存成功，但用户取消订单或支付失败，库存如何恢复？

### 我的回答

1. 记录库存恢复记录  
   - 取消订单时新增恢复记录（状态=处理中）  
   - 发送恢复库存MQ消息

2. 恢复库存流程  

消费端：

- 查询恢复记录状态（幂等）
- 本地事务：DB库存+1 + 更新恢复记录状态
- Redis Lua incr库存

### 更优回答

1. **记录库存恢复日志**

取消订单：

- 新增恢复记录 status=processing
- 发送恢复库存MQ

2. **消费端恢复库存**

根据恢复ID处理：

- DB库存恢复  
  - 状态判断幂等  
  - DB库存+1  
  - 修改恢复记录状态=done

- Redis库存恢复  
  Lua脚本：
  if not exists(recover_id) then
   incr stock
   set recover_id done
   end
   recover_id TTL：1天
#### 3 超时未支付

创建订单时发送 **延迟消息**  
到期自动执行库存恢复。

#### 4 库存对账兜底
定期校验：DB库存 = 初始库存 - 已下单库存 + 恢复库存

若 Redis ≠ DB：重新同步Redis库存。

# 问题11  
## 库存100，请求50万，会允许多少请求进入Redis扣库存？

### 我的回答
1. 首先计算系统可支持的并发量：比如，nginx1台，网关集群3台，服务集群共3台(网关和服务都是用tomcat默认=2000左右最大连接数)，redis集群3台，那么主要压力在服务网关请求数上
2. 测试系统RT，假如是0.5秒
3. 此前已通过nginx限流、网关限流，此时假如还有50万请求，那最大支持并发数=QPS * RT=6000*0.5=3000
4. 考虑到需要预留容量空间20%-30%，最多允许2400个请求进入
---

### 正确设计
秒杀系统 **不应该允许这么多请求进入Redis**。
原因：库存只有100，其余请求一定失败。
正确做法：
1. Nginx限流  
2. 网关限流  
3. 秒杀闸门  
---
### 秒杀闸门
只允许库存几倍请求进入。
示例：库存100，闸门300

用户请求 → 限流 → 秒杀闸门 → Redis扣库存

# 问题12  
## 闸门500，但一个人点击50次怎么办？

### 我的回答

闸门设计 **同一用户重复请求过滤**。

---

### 更优回答

需要两层控制。

---

#### 1 用户请求去重

Redis Key：seckill:user:activity:userId

存在则拒绝请求
---

#### 2 库存闸门

Redis：seckill:gate:activity = 500

Lua：

if gate > 0 then
gate = gate - 1
allow
else
reject
end

效果：

- 防止用户重复请求  
- 控制进入秒杀的请求量

# 问题13  
## 库存100，但只有70人下单成功，剩余30怎么办？

### 我的回答

方案1：多轮秒杀

每轮结束后，剩余库存进入下一轮。

方案2：库存恢复

取消订单：

- 新增库存恢复记录  
- 发送恢复库存MQ  

消费端：

1. Redis Lua incr库存  
2. DB库存+1  
3. 更新恢复记录状态

# 问题14  
## 库存100，50万人抢，有人用脚本创建10万个账号怎么办？

### 我的回答

1. 账号注册控制  
   - 手机号注册  
   - 验证码  
   - 实名认证  

2. 登录鉴权  
   - token校验  
   - 未登录不可抢购  

3. 常用设备校验  
   - 非常用设备二次验证  

---

### 更优回答

多层防刷策略：

1. **账号成本**

- 手机号注册  
- 实名认证  

2. **入口限流**

- IP限流  
- 网关限流  
- 用户限流  

3. **秒杀Token**

流程：

用户进入秒杀页面 → 获取token → 请求秒杀接口

规则：

- token 30秒过期  
- 每个用户一个token  
- token使用后失效  

4. **行为风控**

检测：

- 异常设备  
- 异常频率  
- 同IP大量账号  

5. **动态接口**

示例：/seckill/buy/8f3ab29cd
接口地址动态生成，防止脚本直接调用。

# 秒杀系统整体架构

完整流程：

用户请求 → CDN → Nginx → 网关 → 秒杀Token → 秒杀闸门 → Redis扣库存 → MQ削峰 → 订单服务 → DB落库

---

## 组件职责

| 组件      | 作用                       |
| --------- | -------------------------- |
| CDN       | 静态资源缓存               |
| Nginx     | IP限流                     |
| 网关      | 登录校验、用户限流         |
| 秒杀Token | 防止接口被直接刷           |
| 秒杀闸门  | 控制进入秒杀流程的请求数量 |
| Redis     | 高并发库存扣减             |
| MQ        | 削峰保护数据库             |
| 订单服务  | 创建订单                   |
| DB        | 最终数据存储               |

---

# 防刷策略总结

组合防护：

- 账号成本（手机号、实名）
- 接口限流（IP / 用户）
- 秒杀Token
- 行为风控（设备 / 频率）
- 验证码
