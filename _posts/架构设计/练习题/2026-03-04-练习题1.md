# 秒杀/高并发系统面试练习复盘

---

## 问题1（基础）

**题目**：  
假设当前系统：QPS = 800，RT = 50ms，并发 = 40，数据库连接池 = 100  
10倍流量后：并发 = 400  
数据库连接池会发生什么？正常工作、请求排队、数据库崩溃？请说明原因  

**我的回答**：  
1. 假设请求依次经过网关层->服务层->DB层  
2. 网关层并发=400时，首先判断网关层能否支撑该并发量，若不能，如网关连接池只支持100并发，则后续会进行排队，那么db会正常工作；若超过100但小于400，根据网关配置，其他请求会优先排队获取连接直至超时；若网关未正确配置如拒绝策略会崩溃  
3. 若能支撑该并发量，则全部请求会打到DB层。db此时连接池100，会优先排队，若数据库连接池未正确配置拒绝策略也会崩溃  
4. 若服务层含有缓存，如请求是查询一些热点数据，会提前缓存，则直接请求缓存，不会打到db层，db正常工作  
5. 若服务层含有RPC、请求第三方接口等可能超时服务，会因并发量，根据RPC连接池是否耗尽或排队或崩溃，导致未打到db层。既然问题在问db是否工作，那根据最终实际打到db层的请求量，三种都有可能  

**更优回答**：  
1. 会出现请求排队，RT上升，最终可能导致大量超时。(先给结论，再说原因)  
2. 原因：DB只允许100个请求同时连接，剩余300个会排队等待获取连接，若超时则报错SQLTimeoutException等异常  

**追问**：排队会导致什么？  
- RT(耗时)升高，因为排队时间 + 业务执行时间增加  
- 并发 = QPS * RT，即使QPS不变，RT变大会导致并发量升高  
- 可能导致连接池和线程池耗尽，形成系统雪崩（级联放大效应）  

**解析**：  
我的回答是典型的工程师思维，这里可能，那里可能等各种情况  
架构面试：先给结论 -> 解释原因 -> 推演后果 -> 给解决方案

## 问题2（阿里/字节高频）

**题目**：  
系统QPS = 800，RT = 50ms，突然一次促销活动流量增长10倍，QPS = 8000，如果你只能做一件事来保护系统，你会做什么（增加服务器/接口限流/增加缓存/MQ削峰）？  

**我的回答**：  
接口限流，同时主要是对查询接口限流，而非促销活动下单的主业务接口限流，将资源协调到主业务，保障不受影响。  
首先计算下各服务组件支持的最大连接数，当前网关层并发量到400，DB连接池、线程池大小各是多少。  
若增加服务器，增加缓存，MQ削峰，可能通过其他方式协调就可以满足并发量，不一定要这样解决  

**更优回答**：  
当流量从800QPS增长到8000QPS时，系统各层容量可能无法支撑，最容易出现的风险是数据库连接池耗尽、线程池排队，从而导致RT升高并形成级联雪崩。  
如果只能做一件事，我会优先选择 **接口限流**。  
- 原因：限流可以快速在网关层控制进入系统的请求量，使系统QPS不超过当前系统的承载能力，从而保护数据库和服务不被打垮。  
- 后续：在系统稳定之后，再逐步通过缓存、MQ削峰、分库分表等方式提升系统容量。  

**补充**：这种问题可以先问面试官是突发流量还是预估流量，少量情况下可都回答。

## 问题3（容量规划）

**题目**：  
系统最大承载能力是：3000 QPS。但促销峰值是：8000 QPS  
如果你是架构师，你会把限流值设置为多少，是否会大于3000？  

**我的回答**：  
2500，略低于3000，我的经验是设置在最大QPS的80%左右，是为了给突发情况做预留空间。  
若设置=3000，服务层在执行时若存在网络波动、缓存不稳定、RPC服务延迟等异常情况，会导致RT和并发量也会增加，需要做预留。  
若过低会导致资源空闲，促销峰值导致体验差  

**解析**：  
- 容量规划原则：容量预留、异常波动、资源利用率  
- 补充：最好加一句**动态调整**，根据监控实时QPS、RT指标调整限流，如 Sentinel。

## 问题4（高并发防线）

**题目**：  
系统最大承载能力是：3000 QPS。秒杀活动峰值是：50000 QPS。若只做2500限流，用户疯狂刷新重复请求，系统仍会打爆。  
秒杀系统通常会在哪一层做第一道限流（高并发系统的第一道防线在哪里）？Nginx、网关、业务服务、数据库  

**我的回答**：  
nginx。因为用户请求首先会到达nginx上，且nginx性能高消耗资源少。可以通过限制ip、api路径等方式限流  

**解析**：  
- 限流越靠前越好，避免系统已消耗资源再去限流  
- 实际多层限流：  
  - Nginx/IP路径  
  - 网关/接口级限流  
  - 服务层/线程池/连接池限流  
- CDN缓存可进一步减轻压力

## 问题5（秒杀请求排队实现）

**题目**：  
秒杀系统做nginx限流后，用户疯狂刷新，此时系统会直接将此类大量请求直接返回失败，而不是竞争。是用什么实现的？  
Redis 预扣库存 / 数据库库存扣减 / MQ排队 / JVM锁控制库存  

**我的回答**：  
MQ排队。维护一个可参与秒杀的用户队列，请求后先往队列里塞，若满额则直接返回失败。  
Redis预扣库存、数据库库存扣减已经开始扣减库存了，若逻辑有误很可能会扣减成功导致排队失败  
JVM锁控制库存会完全依赖jvm，消耗服务内存资源  

**更优回答**：  
Redis 预扣库存。Redis库存 > 0，DECR库存，之后进入MQ准备后续逻辑。  
秒杀系统完整流程：CDN -> Nginx限流 -> Gateway限流 -> Redis库存判断 -> MQ削峰 -> 订单服务 -> 数据库

## 问题6（库存原子操作）

**题目**：  
秒杀系统库存100，Redis库存判断也做了，此时Redis库存=1，但同时两个请求进来，如果操作不是原子库存就会变为-1超卖，此时怎么解决？  
Redis 分布式锁 / Redis Lua 脚本 / 数据库事务 / JVM synchronized  

**我的回答**：  
Redis Lua 脚本。原子性操作库存值通过返回值判断是否扣减成功。  
redis分布式锁若锁住库存key：  
- 1是锁会导致响应时间变长，增加资源消耗，体验差  
- 2是redis锁住后，第一个请求拿到锁扣减库存，第二个人再进入锁后需要再次判断redis库存，增加redis操作，比如此时库存因下单取消会增加1，那么实际可以扣减库存，但已判断过库存为0，反而未抢到  

**更优回答**：  
Redis Lua 脚本，单线程原子操作：读取库存 -> 判断库存 -> 扣减库存 -> 返回结果。整个过程不会被其他请求打断。  
分布式锁粒度太大，降低吞吐量，RT升高。

## 问题7（防止重复下单）

**题目**：  
秒杀系统库存100，redis库存也控制住了，如何防止一个用户重复下单？（同一用户连续请求导致购买多件）  
数据库唯一索引 / Redis用户购买标记 / 分布式锁 / 消息队列去重  

**我的回答**：  
Redis用户购买标记。在Redis Lua扣减库存脚本中，同时判断和标记该用户是否已下单。  
若使用db索引，高并发情况下请求都打到db会耗尽连接池资源；  
redis分布式锁粒度太大，会降低吞吐量，导致RT升高；  
消息队列去重是在扣减redis库存之后操作，redis对同一个人已经扣减过几次库存了，此时mq再处理已经晚了，若放这一层是否还要考虑返还库存？增加复杂度

## 总结（复盘）

### 秒杀系统高并发流程（复盘版）
用户 -> CDN -> Nginx限流 -> Gateway限流 -> Redis Lua库存判断 & 用户标记 -> MQ削峰 -> 订单服务 -> DB
### 核心设计原则
1. **限流靠前** → 节省资源  
2. **Redis Lua原子扣减库存** → 避免超卖  
3. **MQ削峰** → 平滑处理请求，保护DB  
4. **数据库最终一致性** → Redis + MQ + DB配合  
5. **用户重复购买防护** → Redis用户标记  

### 面试回答顺序
**结论 → 原因 → 推演后果 → 解决方案**  
- 可区分突发流量 vs 预期流量  
- 短期先保护系统，长期逐步扩容
